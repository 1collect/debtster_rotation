# Rotation Go

API-сервис для перераспределения материалов из XLSX-файлов между логинами. Приложение принимает Excel-файл, запускает один из режимов обработки, показывает прогресс через WebSocket и отдает готовый XLSX с заполненной колонкой `Закрепить` и листом итогов.

Интерфейс и серверные сообщения рассчитаны на русскоязычного пользователя.

## Что умеет

- Загружает `.xlsx` через HTTP API.
- Выполняет три основных режима:
  - `Ротация потоками` - полное перераспределение внутри каждого РП.
  - `Ротация между РП` - перераспределение внутри и между РП с сохранением близкого количества материалов по каждому РП.
  - `Выравнивание` - частичное выравнивание текущего закрепления.
- Авторизует по сервисному токену или Bearer-токену пользователя.
- Показывает live-прогресс и подробный лог этапов.
- Пишет подробные server-side логи в stdout контейнера.
- Держит runtime-состояние задач и XLSX-результат в памяти процесса.
- Позволяет отменить текущую задачу.
- Очищает текущую in-memory историю через API.

## Быстрый запуск

Создайте `.env` в корне проекта:

```env
SERVICE_TOKEN=change-me-to-a-long-random-token
```

Запустите сервис:

```bash
sudo docker compose up -d --build --force-recreate
```

API будет доступен на:

```text
http://localhost:8040
```

Порт `8040` задан в `docker-compose.yml`:

```yaml
network_mode: host
environment:
  - ADDR=:8040
```

Compose использует host-сеть, поэтому приложение слушает `8040` напрямую на хосте.

## Логи

Смотреть логи:

```bash
sudo docker compose logs -f
```

При старте должно быть:

```text
listening on :8040
```

На каждый HTTP-запрос сервер пишет access-log:

```text
http request method=GET path=/ remote=...
http done method=GET path=/ elapsed=...
```

При запуске обработки появляются диагностические строки:

```text
upload accepted process=rotation job=... file="..." bytes=...
job started process=rotation job=... strategy=full_parallel source_column=detach
workbook diagnostics job=... rows=... missing_iin=... missing_login=... fixed_rows=... rotation_rows=...
job progress job=... percent=... message="..."
```

Если в логах нет `http request`, запрос отправлен не на тот порт или не на этот контейнер. Для текущего compose нужен `http://localhost:8040`.

## Формат Excel

Обрабатывается первый лист книги. Заголовки должны быть в первой строке.

Обязательные колонки:

| Колонка | Для чего используется |
| --- | --- |
| `РП` | Группа/портфель. В режиме `Ротация между РП` значение может быть заменено на целевой РП. |
| `ИИН` | Идентификатор должника. Строки группируются по ИИН. |
| `Общая задолженность...` | Сумма задолженности. Подходит колонка, название которой начинается с `Общая задолженность`. |
| `Открепить` | Исходный логин для ротации. |
| `Закрепить` | Целевой логин. В эту колонку записывается результат. Для выравнивания это текущий логин. |
| `Статус` | Статус материала. Некоторые статусы фиксируются и не перераспределяются. |

Повторные строки заголовков внутри файла пропускаются.

Суммы читаются в разных форматах:

- `1234567.89`
- `1 234 567,89`
- `1.234.567,89`
- `1,234,567.89`
- `5.6888806E+06`
- `5,6888806E+06`

Расчеты выполняются через `decimal`.

## Режимы

### Ротация потоками

Endpoint: `POST /rotate-parallel/`

Что делает:

- берет логины из `Открепить`;
- группирует строки по `РП + ИИН`;
- распределяет группы между логинами внутри каждого РП;
- заполняет `Закрепить`;
- создает лист `Итоги параллельной ротации`.

### Ротация между РП

Endpoint: `POST /rotate-between-rp/`

Что делает:

- берет логины из `Открепить`;
- группирует строки по исходным `РП + ИИН`;
- распределяет группы по логинам всех РП;
- сохраняет близкое количество материалов по каждому РП: сколько строк ушло из РП, примерно столько же должно прийти обратно;
- записывает исходное значение РП в колонку `Исходное рп`;
- обновляет `РП` на РП выбранного логина и заполняет `Закрепить`;
- создает лист `Итоги ротации между РП`.

### Выравнивание

Endpoint: `POST /balance/`

Что делает:

- берет текущие логины из `Закрепить`;
- выравнивает нагрузку внутри каждого РП;
- двигает только часть ИИН;
- создает лист `Итоги выравнивания`.

## Фиксированные статусы

Для режимов ротации фиксируются:

- `Оплата ПВ`
- `Оплата по соглашению`
- `Оплата по первому условию`
- `Оплата по второму условию`
- `Полное погашение`
- `Полное погошение`

Для выравнивания дополнительно фиксируется:

- `Должник обещает оплату`

Фиксированные строки остаются за текущим логином и попадают в итоговую нагрузку.

## Результат

Готовый XLSX содержит:

- исходный первый лист с заполненной или обновленной колонкой `Закрепить`;
- для режима `Ротация между РП` колонку `Исходное рп` с РП строки до перераспределения;
- отдельный лист итогов.

Итоговый лист содержит:

| Колонка | Описание |
| --- | --- |
| `РП` | РП логина. |
| `Логин` | Логин исполнителя |
| `Количество материалов` | Количество строк |
| `Сумма задолженности` | Суммарная задолженность |
| `Количество ИИН` | Количество уникальных ИИН |

Внизу листа итогов добавляются:

- количество строк, зафиксированных по статусам оплаты;
- количество ИИН, зафиксированных по статусам оплаты.

## API

Рабочие endpoints требуют `X-Rotation-Token`, `Authorization: Bearer ...` или `token` в query string.

`/upload` принимает Bearer/Sanctum-токен пользователя, потому что сервер сохраняет `user_id` в MongoDB.

### Авторизация

Для сервисного доступа передавайте значение `SERVICE_TOKEN`:

```http
X-Rotation-Token: change-me-to-a-long-random-token
```

Для пользовательского доступа можно передать Bearer-токен основной системы:

```http
Authorization: Bearer <sanctum-token>
```

### Обработка файлов

Все endpoints принимают `multipart/form-data`:

- `file` - XLSX-файл;
- `job_id` - необязательный идентификатор задачи. Если не передан, сервер сгенерирует его сам.

Endpoints:

- `POST /rotate-parallel/`
- `POST /rotate-between-rp/`
- `POST /balance/`

Успешный ответ:

```json
{
  "job_id": "...",
  "state": "running"
}
```

### Задачи

`GET /api/jobs`

Возвращает список задач.

`GET /api/jobs/{job_id}`

Возвращает задачу вместе с логом прогресса.

`POST /api/jobs/{job_id}/cancel`

Отменяет текущую задачу.

`POST /api/history/clear`

Отменяет выполняющиеся задачи, очищает историю и удаляет XLSX-результаты.

`GET /download/{job_id}/`

Скачивает результат готовой задачи.

### WebSocket

`GET /ws/rotation/{job_id}/`

Сообщение прогресса:

```json
{
  "type": "progress",
  "percent": 48,
  "message": "Считаю нагрузку по логинам и подбираю распределение без учета РП"
}
```

Готово:

```json
{
  "type": "job_ready",
  "message": "Файл готов.",
  "download_url": "/download/{job_id}/"
}
```

Ошибка:

```json
{
  "type": "job_error",
  "message": "Описание ошибки"
}
```

Отмена:

```json
{
  "type": "job_canceled",
  "message": "Задача отменена."
}
```

## Хранение данных

Go-сервис не пишет историю и XLSX-результаты на локальный диск. Текущее состояние задач, лог прогресса и готовый файл живут в памяти процесса. При настроенных Mongo/MinIO интеграциях запись ротации обновляется в Mongo, а готовый XLSX сохраняется в MinIO.

Если сервер перезапущен во время обработки, in-memory состояние и временный download endpoint теряются. Постоянным источником статуса остается запись в Mongo при включенной интеграции.

## Ограничения

- Одновременно выполняется только одна задача.
- Поддерживается только `.xlsx`.
- Обрабатывается только первый лист.
- Заголовки должны быть в первой строке.
- In-memory история и временный download endpoint сбрасываются после перезапуска сервера.

## Частые проблемы

### После запуска в логах только `listening on :8040`

Это нормально: сервер ждет запросы. Отправьте запрос на:

```text
http://localhost:8040
```

После запроса в логах должны появиться `http request`.

### Открывается не тот порт

Текущий compose использует host-сеть и слушает:

```text
:8040
```

Проверить:

```bash
sudo docker compose ps
```

### Приложение пишет, что уже идет операция

Одновременно разрешена только одна задача. Дождитесь завершения или нажмите `Отменить`.

### Файл не принимается

Проверьте:

- расширение `.xlsx`;
- обязательные колонки;
- первый лист;
- заголовки в первой строке;
- наличие ИИН и логинов в рабочих строках.

Диагностика по строкам пишется в логи как `workbook diagnostics`.

### После перезапуска задача стала отмененной

Это ожидаемо. Выполняющаяся задача не продолжается после остановки процесса.

## Безопасность

- Задайте свой `SERVICE_TOKEN`.
- Не используйте дефолтный `debtster-rotation` в рабочем запуске.
- При публикации наружу ставьте reverse proxy с HTTPS.
- Ограничьте доступ к MongoDB и MinIO/S3, потому что там лежат статусы и файлы обработки.
